# Metrics Dictionary

The business question this project answers: **are we acquiring players
profitably?** The headline metric is **D30 ROAS** (and its sibling, the
**LTV : CAC** ratio). Everything else is a driver of, or context for, that
number.

Golden rule (enforced in `models/marts/README.md` and the Semantic Layer):
**never `AVG()` a rate** — store the numerator and denominator and recompute.

---

## Headline metric

### ROAS — Return On Ad Spend
- **Definition:** revenue generated by acquired users ÷ ad spend.
- **SQL logic:** `SAFE_DIVIDE(SUM(cohort_revenue), NULLIF(SUM(spend), 0))`
  from `ua_performance_daily` (Adjust is the only source that joins spend to
  attributed revenue). Semantic Layer: `metric: roas` (ratio of
  `cohort_revenue` / `spend`).
- **Business impact:** ROAS ≥ 1 means a campaign pays back within the cohort
  window. It is the primary signal for scaling or cutting UA budget by
  network/campaign/geo.

### LTV : CAC
- **Definition:** projected per-user lifetime value ÷ cost to acquire that user.
- **SQL logic:** LTV(N) from `cohort_ltv_daily`
  `SAFE_DIVIDE(SUM(IF(days_since_install <= N, iap_revenue, 0)), ANY_VALUE(cohort_size))`;
  CAC = eCPI from `ua_performance_daily`. Ratio of the two.
- **Business impact:** the long-horizon profitability view ROAS can't see early;
  drives payback-period decisions.

## Acquisition economics (`ua_performance_daily`, `ua_funnel_daily`)

| Metric | Definition | SQL logic |
|---|---|---|
| **Spend** | Ad-network cost | `SUM(cost)` |
| **Installs** | Attributed installs | `SUM(installs)` (+ `paid`/`organic` splits) |
| **eCPI / CPI** | Cost per install | `SAFE_DIVIDE(SUM(spend), NULLIF(SUM(installs),0))` |
| **CTR** | Click-through rate | `SAFE_DIVIDE(SUM(clicks), NULLIF(SUM(impressions),0))` |
| **CVR** | Click→install rate | `SAFE_DIVIDE(SUM(installs), NULLIF(SUM(clicks),0))` |
| **ROI** | Net return on spend | `SAFE_DIVIDE(SUM(all_revenue)-SUM(cost), NULLIF(SUM(cost),0))` |
| **Funnel rates** | Impr→click→install→purchase | stage ratios in `ua_funnel_daily` |

**Impact:** eCPI is the price of a player; the funnel localises *where* a
campaign leaks (creative vs store page vs onboarding).

## Monetisation (`engagement_daily`, `iap_arpdau_daily`, `ua_performance_daily`)

| Metric | Definition | SQL logic |
|---|---|---|
| **ARPDAU** | Avg revenue / daily active user | `SAFE_DIVIDE(SUM(revenue), NULLIF(SUM(dau),0))` |
| **ARPPU** | Avg revenue / paying user | `SAFE_DIVIDE(SUM(iap_revenue), NULLIF(SUM(payers),0))` |
| **IAP revenue** | In-app-purchase revenue | `SUM(iap_value_usd())` over IAP events only — see `macros/iap.sql` |
| **IAP vs ad split** | Revenue mix | `iap_share`, `ad_share` in `ua_performance_daily` |

**Scope matters** (documented in the marts README): *Unity* ARPDAU = whole game;
*Adjust* ARPDAU = paid-UA users only; *Meta* ROAS = Meta's own attribution.
These are labelled separately and are not expected to match.

> The IAP value definition deliberately **excludes the generic `amount` event
> param** (it carries in-game coin amounts, not USD). Centralised in
> `iap_value_usd()` so all marts agree — this fixed the historic
> `retention_cohorts` inflation bug.

## Engagement & retention (`engagement_daily`, `retention_cohorts`, `cohort_ltv_daily`)

| Metric | Definition | SQL logic |
|---|---|---|
| **DAU / WAU / MAU** | Active users | `COUNT(DISTINCT user_pseudo_id)` (Firebase) / Unity rollups |
| **DN retention** | % of cohort active N days post-install | `retained_dN_users / cohort_size` |
| **LTV(N)** | Cumulative IAP per installed user by day N | `SUM(iap_revenue WHERE dsi<=N) / cohort_size` |
| **Session depth** | Events / duration / levels per session | avg + p25..p99 in `session_depth_daily` |
| **Completion rate** | Tutorial finishers ÷ starters | `tutorial_funnel_daily` (step_num NULL ÷ step_num=1) |

**Impact:** retention is the leading indicator of LTV; if D1/D7 drop, ROAS will
follow weeks later. Session depth and tutorial completion explain *why* and feed
product fixes (difficulty spikes, onboarding drop-off).
